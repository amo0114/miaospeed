 # MiaoSpeed 前端扩展与优化建议

 > 范围说明：本文只讨论前端扩展与优化，不包含后端功能扩展设计。
 >
 > 目标：基于当前 WebUI 结构，梳理后续可扩展功能、落地方向，以及还值得继续优化改进的部分。

 ---

 ## 1. 文档目的

 当前前端已经具备基础的：

 - 节点导入
 - 连接配置
 - WebSocket 交互
 - 测试发起
 - 进度显示
 - 结果表格展示

 但从后续演进看，当前前端仍然偏“能跑流程”的版本，还没有真正成长为：

 - 易联调
 - 易定位问题
 - 易扩展功能
 - 易维护演进

 因此，这份文档分成两部分：

 1. 哪些功能值得继续扩展。
 2. 哪些现有设计还值得进一步优化。

 ---

 ## 2. 当前前端能力现状

 从当前代码结构看，前端主要由以下模块组成：

 - `web/src/App.tsx`
   - 页面装配
   - 配置持久化
   - 测试预设转发
 - `web/src/hooks/use-miaospeed.ts`
   - WebSocket 连接
   - 消息解析
   - 请求发送
   - 结果转换
 - `web/src/lib/crypto/sign.ts`
   - 请求构建与签名
 - `web/src/components/node-importer.tsx`
   - 节点导入
 - `web/src/components/test-panel.tsx`
   - 测试项选择与触发
 - `web/src/components/results-table.tsx`
   - 最终结果展示
 - `web/src/components/settings-dialog.tsx`
   - 连接设置

 当前架构的优点：

 - 已经把 WebSocket 逻辑集中在 hook 里
 - 已有较完整的 TypeScript 类型定义
 - UI 结构清晰，组件职责大体明确

 当前架构的限制：

 - 配置来源分散
 - 测试预设重复定义
 - 连接模式没有抽象清楚
 - 结果展示仍偏静态
 - 调试与诊断能力不足

 这些限制决定了：

 - 前端后续扩展不能只堆 UI
 - 应该优先补齐“配置、连接、诊断、结果流”这四条基础能力

 ---

 ## 3. 推荐扩展功能

 ## 3.1 联调辅助功能

 这是最值得优先扩展的一类，因为它直接降低联调成本。

 ### 功能 A：连接诊断面板

 **功能价值**

 让前端直接显示当前联调上下文，避免开发者靠猜。

 **建议展示内容**

 - 当前最终 WebSocket 地址
 - 当前 `serverUrl`
 - 当前 `wsPath`
 - 当前是否走代理模式
 - 当前 Startup Token 是否为空
 - 当前 Build Token 来源（本地配置 / env / 默认值）
 - 最近一次连接错误
 - 最近一次请求是否验签成功

 **建议落点**

 - 新增一个小型调试组件
 - 可放在设置弹窗中，或做成可折叠调试卡片

 **优先级**

 - 高

 **复杂度**

 - 低到中

 ### 功能 B：配置重置与模式切换

 **功能价值**

 当前最容易困扰联调的是：本地缓存配置、代理模式、直连模式混在一起。增加明确切换入口后，联调路径会清晰很多。

 **建议功能点**

 - 一键清空 `localStorage` 配置
 - 一键恢复默认配置
 - 预置两种模式：
   - 本地代理模式
   - 直连后端模式

 **优先级**

 - 高

 **复杂度**

 - 中

 ### 功能 C：请求预览与签名状态

 **功能价值**

 帮助开发者在发送前看到：

 - 实际请求 JSON
 - 参与签名的关键字段
 - 当前 Challenge 是否已生成

 **建议功能点**

 - “预览本次请求”按钮
 - 显示生成后的 `Basics`、`Matrices`、`Configs`、`Challenge`
 - 支持复制 JSON 用于排查

 **优先级**

 - 高

 **复杂度**

 - 中

 ---

 ## 3.2 测试体验增强功能

 ### 功能 D：统一且可扩展的测试预设

 **功能价值**

 当前预设重复定义且已出现不一致。后续应该把测试预设变成一个真正可扩展的前端能力，而不是散落常量。

 **建议功能点**

 - 统一预设定义来源
 - 支持新增“轻量测速 / 完整测速 / 自定义测速”
 - 支持为每个预设配置：
   - matrices
   - 下载参数
   - 上传参数
   - timeout

 **优先级**

 - 高

 **复杂度**

 - 中

 ### 功能 E：实时结果流入主表格

 **功能价值**

 当前用户只能在进度区看到过程，在结果表里看到最终状态。更好的体验应该是：测试过程中主表格就逐步刷新。

 **建议功能点**

 - `Progress.Record` 直接更新结果表格
 - 已完成项与排队项在表格中分层显示
 - 支持显示“测试中”状态

 **优先级**

 - 高

 **复杂度**

 - 中

 ### 功能 F：停止测试与重新测试

 **功能价值**

 当前前端更偏一次性发起测试，缺少中止、重试、重测单个节点的能力。

 **建议功能点**

 - 停止当前测试
 - 重测失败节点
 - 按当前排序结果重测选中节点

 **优先级**

 - 中

 **复杂度**

 - 中到高

 ---

 ## 3.3 结果展示增强功能

 ### 功能 G：结果筛选与搜索

 **功能价值**

 当前结果多了以后，单纯排序不够用。至少应该能按国家、协议、延迟范围筛选。

 **建议功能点**

 - 按节点名搜索
 - 按代理协议筛选
 - 按国家/GeoIP 筛选
 - 按 RTT/速度范围筛选

 **优先级**

 - 中

 **复杂度**

 - 中

 ### 功能 H：结果视图切换

 **功能价值**

 有的用户偏表格，有的用户更适合卡片、分组、统计视图。

 **建议功能点**

 - 表格视图
 - 协议分组视图
 - 国家分组视图
 - 统计摘要视图

 **优先级**

 - 中

 **复杂度**

 - 中到高

 ### 功能 I：历史测试记录与结果对比

 **功能价值**

 这是前端非常有价值的增强点，因为很多用户并不是只看一次测速，而是要比较不同时间、不同配置、不同订阅的结果。

 **建议功能点**

 - 保存最近 N 次测试结果到本地
 - 结果对比视图
 - 导出 JSON / CSV / Markdown

 **优先级**

 - 中

 **复杂度**

 - 中到高

 ---

 ## 3.4 配置管理增强功能

 ### 功能 J：多套连接配置档案

 **功能价值**

 当前只有一套配置，不利于在不同环境间切换。

 **建议场景**

 - 本地开发代理模式
 - 本地直连模式
 - 测试服务器
 - 生产服务器

 **建议功能点**

 - 配置档案列表
 - 新建/复制/删除档案
 - 一键切换当前档案

 **优先级**

 - 中

 **复杂度**

 - 中

 ### 功能 K：配置导入导出

 **功能价值**

 便于团队共享联调配置，也便于用户备份。

 **建议功能点**

 - 导出当前配置 JSON
 - 导入配置 JSON
 - 导入时校验字段完整性

 **优先级**

 - 中

 **复杂度**

 - 低到中

 ---

 ## 3.5 可观测性与调试能力增强

 ### 功能 L：前端事件日志面板

 **功能价值**

 让开发者看到关键事件顺序，尤其适合联调排查。

 **建议记录事件**

 - 配置加载
 - 连接开始
 - 连接成功/失败
 - 收到错误帧
 - 发起请求
 - 收到进度
 - 收到最终结果

 **优先级**

 - 高

 **复杂度**

 - 中

 ### 功能 M：错误分类与可读提示

 **功能价值**

 现在很多错误仍然是原始文本。前端应把错误转成更可理解的提示层。

 **建议分类**

 - 路径错误
 - Token/签名错误
 - TLS/协议错误
 - 节点解析错误
 - 请求构建错误

 **优先级**

 - 高

 **复杂度**

 - 中

 ## 4. 当前前端可优化项

 ## 4.1 架构优化

 ### 优化 O1：统一配置入口

 当前配置分散在：

 - `DEFAULT_CONFIG`
 - `.env`
 - `localStorage`
 - SettingsDialog
 - `useMiaoSpeed`

 建议后续至少把配置逻辑收敛成两个明确层次：

 1. 配置解析层
 2. 配置消费层

 推荐方向：

 - 新增 `useMiaoSpeedConfig` 或 `lib/config` 模块
 - 统一负责默认值、环境变量、本地持久化、运行时配置合并

 这样后续新增配置项时，不会到处补洞。

 ### 优化 O2：统一测试预设入口

 预设应从 UI 组件和 App 逻辑中抽离，成为单一数据源。

 推荐方向：

 - 新增 `web/src/lib/test-presets.ts`
 - UI 只读预设
 - 请求构建也只读同一份预设

 ### 优化 O3：把协议构建与 UI 状态继续解耦

 当前 `useMiaoSpeed.ts` 同时承担：

 - 连接管理
 - 错误处理
 - 结果解析
 - 请求发送

 后续功能增多时，可以继续拆成：

 - 连接层
 - 请求构建层
 - 响应解析层
 - UI 适配层

 不建议现在就上复杂状态库，但建议先把职责边界划清。

 ---

 ## 4.2 稳定性优化

 ### 优化 O4：配置加载加保护

 当前 `localStorage` 解析建议增加：

 - try/catch
 - 字段兜底
 - 版本迁移

 这样以后调整配置结构时，不会因为旧配置而影响页面启动。

 ### 优化 O5：对节点导入与请求构建做更清晰校验

 推荐补充：

 - 节点为空校验
 - 节点类型识别失败提示
 - 请求矩阵为空校验
 - Token 未配置提示

 ### 优化 O6：连接状态更精细化

 当前状态值较简单。后续可以细分成：

 - disconnected
 - connecting
 - connected
 - authenticating
 - running
 - completed
 - error

 这样更有利于做 UI 提示和调试日志。

 ---

 ## 4.3 UX 优化

 ### 优化 O7：设置变更后的重连提示

 当前保存设置后，用户不一定意识到是否需要重连。建议至少补其中一个：

 - 配置变更后显示“需重新连接”提示
 - 若当前已连接，保存后弹确认：是否立即断开并重连

 ### 优化 O8：空状态与错误状态更可读

 当前视觉已经不错，但仍可继续增强：

 - 区分“未连接”“无结果”“测试中”“连接失败”
 - 为不同错误提供具体建议，而不是只回显原始文本

 ### 优化 O9：敏感配置显示策略

 Token 和 Build Token 的显示可以更谨慎：

 - 默认掩码
 - 支持查看/隐藏
 - 支持一键复制
 - 支持校验格式

 ---

 ## 4.4 工程质量优化

 ### 优化 O10：补测试

 建议优先补的前端测试点：

 - 签名算法输出一致性
 - 测试预设映射一致性
 - 节点清空后父状态同步
 - 配置持久化与恢复
 - WebSocket 消息解析

 ### 优化 O11：补文档同步机制

 后续每次改动以下内容时，应该同步检查文档：

 - 默认连接地址
 - 路径
 - TLS 策略
 - env 变量名
 - 包版本

 否则文档会再次快速漂移。

 ### 优化 O12：给前端 README 一个明确定位

 当前 `web/README.md` 更像模板遗留。建议后续改为：

 - 如何启动前端
 - 本地联调有哪两种模式
 - 当前推荐模式是什么
 - 常见错误与排查方式

 让新开发者第一眼就看到真正有用的信息。

 ---

 ## 5. 中短期迭代建议

 ## 5.1 短期建议（优先做）

 这些建议应与联调修复同步推进：

 1. 增加连接诊断面板
 2. 增加配置重置与模式切换
 3. 抽离测试预设单一来源
 4. 让实时结果进入主表格
 5. 增加错误分类与可读提示

 这几个改动收益大、复杂度可控，而且能直接提升当前前端的“可调试性”。

 ## 5.2 中期建议（联调稳定后做）

 1. 多套连接配置档案
 2. 配置导入导出
 3. 历史测试记录
 4. 结果对比视图
 5. 筛选/搜索/分组视图

 这些功能会让前端从“联调界面”升级成“可长期使用的测速工作台”。

 ## 5.3 暂缓项

 以下方向不是不能做，而是没必要现在抢做：

 - 复杂状态管理库全面接管
 - 大规模图表系统
 - 过度抽象的插件架构
 - 与后端深度耦合的新协议扩展 UI

 原因很简单：

 - 当前最缺的是基础一致性和可观测性
 - 不是大型平台级抽象能力

 ## 6. 功能扩展优先级建议

 | 类别 | 功能 | 优先级 | 复杂度 | 说明 |
 | --- | --- | --- | --- | --- |
 | 联调辅助 | 连接诊断面板 | 高 | 低到中 | 最能直接降低排查成本 |
 | 联调辅助 | 配置重置与模式切换 | 高 | 中 | 解决本地缓存与模式混乱 |
 | 联调辅助 | 请求预览与签名状态 | 高 | 中 | 对签名问题定位价值很高 |
 | 测试体验 | 统一测试预设 | 高 | 中 | 是后续扩展的基础 |
 | 测试体验 | 实时结果进入主表格 | 高 | 中 | 体验提升明显 |
 | 可观测性 | 前端事件日志 | 高 | 中 | 适合联调阶段长期保留 |
 | 可观测性 | 错误分类与提示 | 高 | 中 | 提升理解成本与排障效率 |
 | 结果增强 | 搜索/筛选 | 中 | 中 | 节点多时价值明显 |
 | 配置增强 | 多套配置档案 | 中 | 中 | 多环境切换更方便 |
 | 结果增强 | 历史结果对比 | 中 | 中到高 | 使用价值高，但依赖更多状态设计 |
 | 体验增强 | 停止/重测 | 中 | 中到高 | 需处理更多运行状态 |
 | 展示增强 | 统计视图/图表 | 低到中 | 中到高 | 不是当前最急需 |

 ---

 ## 7. 结论

 这个前端后续完全可以继续演进，但不建议直接堆功能。

 更合理的路线是：

 1. 先把联调能力做扎实
 2. 再把连接与诊断能力做透明
 3. 再把测试预设和结果展示做强
 4. 最后才做历史、对比、配置档案等工作台能力

 当前最值得优先扩展的，不是更复杂的可视化，而是：

 - **更清楚地告诉用户“现在连到了哪里、为什么失败、发了什么请求、拿回了什么结果”**

 一旦这层能力补齐，前端再扩展任何高级功能，都会顺畅很多。
